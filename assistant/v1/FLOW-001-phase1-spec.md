### 1-2. 플로우 업무 상태 정의

브리핑에서 참조하는 플로우 업무 상태는 다음과 같다.

| 상태 | 설명 | 브리핑 노출 여부 |
|------|------|-----------------|
| 요청 | 업무가 할당되었으나 아직 착수하지 않은 상태 | 노출 (행동 유도 대상) |
| 진행중 | 업무를 수행 중인 상태 | 노출 |
| 완료 | 업무가 완료 처리된 상태 | 마무리 브리핑의 완료 업무 요약에서만 노출 |

## 2. 브리핑별 상세 스펙

---

### 2-1. 오전 브리핑

#### 개요

하루 업무 시작 시점에 오늘의 일정, 처리해야 할 업무, 부재중 변화사항을 요약하여 제공하는 브리핑.

---

#### 트리거 조건

| 구분 | 내용 |
|------|------|
| 트리거 유형 | 예약 실행형 (Scheduled) |
| 기본 발송 시각 | 오전 08:00 (사용자 설정 변경 가능) |
| 생성 시작 시각 | 발송 시각 30분 전 (기본값: 07:30) |
| 발송 조건 | 해당 날짜에 사용자의 일정 또는 업무가 1건 이상 존재할 것 |
| 미발송 조건 | 사용자가 오전 브리핑 알림을 OFF로 설정한 경우 |
| 발송 주기 | 매일 1회 (주말 포함, 사용자 설정으로 제외 가능) |

---

#### 데이터 수집 범위 및 쿼리 조건

**[섹션 A] 오늘 일정**

| 항목 | 조건 |
|------|------|
| 수집 대상 | 사용자가 참여자(주최자/초대받은 자)인 캘린더 일정 |
| 날짜 범위 | 브리핑 발송일 00:00 ~ 23:59 (당일 전체) |
| 정렬 기준 | 시작 시각 오름차순 |
| 포함 항목 | 일정 제목, 시작/종료 시각, 참석자 목록, 장소 또는 링크 |
| 제외 조건 | 브리핑 생성 시점 이전에 이미 종료된 일정은 표시하되 '완료' 레이블 부착 |
| 최대 노출 수 | 제한 없음 (전체 표시) |

**[섹션 B] 내가 해야할 업무**

사용자가 담당자로 지정된 업무. 직접 처리·실행해야 하는 업무.

| 항목 | 조건 |
|------|------|
| 수집 대상 | 사용자가 **담당자**인 플로우 업무 |
| 마감일 조건 | 마감일이 당일인 업무 OR 마감일이 1~2일 지난 업무(지연 레이블 부착) OR 마감일이 없고 상태가 '요청' 또는 '진행중'인 업무 |
| 상태 조건 | '완료' 상태 제외 ('요청', '진행중' 포함) |
| 정렬 기준 | 1순위: 마감일 있는 업무 오름차순, 2순위: 상태 '요청' 우선 (미착수), 3순위: 우선순위 높은 순 |
| 포함 항목 | 업무 제목, 마감일, 상태, 프로젝트명, 우선순위 |
| 최대 노출 수 | 최대 10건 (초과 시 "외 N건" 축약 표시) |
| AI 톤 | 행동 유도형 — "~를 먼저 처리하는 게 좋겠어요", "아직 미착수 상태에요" |

**[섹션 B-1] 내가 확인해야할 업무**

사용자가 작성자(요청자)이고 다른 사람이 담당자로 지정된 업무. 진행 상황을 추적·확인해야 하는 업무.

| 항목 | 조건 |
|------|------|
| 수집 대상 | 사용자가 **작성자**이고 다른 사람이 **담당자**인 플로우 업무 |
| 마감일 조건 | 마감일이 당일 OR 익일인 업무 |
| 상태 조건 | '완료' 상태 제외 |
| 정렬 기준 | 1순위: 마감일 오름차순, 2순위: 상태 '요청' 우선 (미착수 = 확인 필요), 3순위: 우선순위 높은 순 |
| 포함 항목 | 업무 제목, 마감일, 상태, 담당자명, 프로젝트명, 우선순위 |
| 최대 노출 수 | 최대 5건 (초과 시 "외 N건" 축약 표시) |
| AI 톤 | 확인 유도형 — "~님에게 요청한 업무가 아직 미착수 상태에요", "진행 상황을 확인해보세요" |
| 데이터 없을 시 | 섹션 비표시 |

**[섹션 B-2] 지연 업무 리마인드**

| 항목 | 조건 |
|------|------|
| 수집 대상 | 사용자가 담당자인 플로우 업무 중 마감일이 지난 업무 |
| 상태 조건 | '완료' 상태 제외 ('요청', '진행중' 포함) |
| 지연 기준 | 마감일이 현재 날짜보다 이전인 업무 |
| 노출 조건 | 지연 3일 이상인 경우에만 리마인드 노출 (1~2일 지연은 섹션B에 포함) |
| 정렬 기준 | 지연 일수 내림차순 (오래된 것 우선) |
| 포함 항목 | 업무 제목, 마감일, 지연 일수, 상태, 프로젝트명 |
| 최대 노출 수 | 최대 3건 개별 표시. 초과 시 "외 N건의 지연 업무가 있어요. 확인하기" → 플로우 업무 목록 연결 |
| 노출 상한 | 지연 업무 10건 이상이면 개별 목록 없이 요약만 표시: "마감이 지난 업무가 N건 있어요. 이미 완료했다면 상태를 바꿔주세요." |
| CTA | "완료했다면 상태 업데이트하기" → 플로우 업무 목록 연결 |
| 데이터 없을 시 | 섹션 비표시 |

**[섹션 C] 부재중 변화**

| 항목 | 조건 |
|------|------|
| 수집 대상 | 직전 브리핑 발송 시각 이후 ~ 현재 사이에 발생한 변화 |
| 변화 유형 | 새로 할당된 업무, 나에게 온 멘션(읽지 않은 것), 담당 업무 상태 변경 |
| 정렬 기준 | 발생 시각 내림차순 (최신순) |
| 포함 항목 | 변화 유형, 업무/메시지 제목, 발생 시각, 작성자/변경자 |
| 최대 노출 수 | 유형별 최대 5건, 전체 최대 10건 |
| 최초 실행 기준 | 직전 브리핑이 없는 경우, 전날 마지막 접속 이후 발생한 변화를 수집 |

---

#### 브리핑 구성 섹션

```
[오전 브리핑 구조]

1. 인삿말 + 날짜 안내
   예: "좋은 아침이에요, {이름}님. 4월 15일 화요일 오늘의 업무를 정리했어요."

2. 오늘 일정 (섹션 A)
   - 일정 없을 경우: "오늘 등록된 일정이 없어요."
   - 일정 있을 경우: 시간순 목록 표시

3. 오늘 해야 하는 업무 (섹션 B)
   - 업무 없을 경우: "마감이 임박한 업무가 없어요."
   - 업무 있을 경우: 우선순위순 목록 표시

4. 지연 업무 리마인드 (섹션 B-2)
   - 지연 3일 이상 업무가 없을 경우: 섹션 비표시
   - 3건 이하: 개별 업무 목록 + "완료했다면 상태를 업데이트해주세요."
   - 4~9건: 최대 3건 표시 + "외 N건" 축약
   - 10건 이상: "마감이 지난 업무가 N건 있어요. 이미 완료했다면 상태를 바꿔주세요."

5. 부재중 변화 (섹션 C)
   - 변화 없을 경우: 섹션 비표시 (섹션 자체 숨김)
   - 변화 있을 경우: 변화 항목 목록 표시

6. 추천 행동 (CTA)
   - 가장 우선순위 높은 업무 또는 첫 번째 일정 기반으로 1~2개 행동 제안
   - 예: "지금 바로 [업무명] 확인하기" / "[일정명] 참여 준비하기"
```

#### 빈 데이터 처리

| 섹션 | 데이터 없음 처리 방식 |
|------|----------------------|
| 오늘 일정 | "오늘 등록된 일정이 없어요." 문구 표시, 섹션 유지 |
| 오늘 업무 | "마감이 임박한 업무가 없어요." 문구 표시, 섹션 유지 |
| 지연 업무 리마인드 | 지연 3일 이상 업무가 없으면 섹션 전체 숨김 |
| 부재중 변화 | 섹션 전체 숨김 처리 |
| 전체 데이터 없음 | 브리핑 생성하되 "오늘은 특별히 정리할 내용이 없어요. 편안한 하루 되세요." 표시 |

### 2-2. 일정 브리핑

#### 개요

캘린더 일정 시작 전, 해당 일정에 관해 미리 참고하면 좋을 사항을 AI가 분석하여 제공하는 브리핑.  
일정 제목·본문 키워드와 참석자 데이터를 기반으로 연관된 플로우 업무·게시글·위키 등을 탐색하고, 필요 시 웹 검색도 수행하여 종합적인 준비 정보를 제공한다.

---

#### 트리거 조건

| 구분 | 내용 |
|------|------|
| 트리거 유형 | 예약 실행형 (Scheduled, 이벤트 기반) |
| 발송 시각 | 일정 시작 시각 기준 N분 전 (기본값: 10분 전, 사용자 설정 가능) |
| 생성 시작 시각 | 발송 시각 30분 전 |
| 발송 조건 | 사용자가 참여자인 일정이 존재할 것 / 일정 시간이 30분 이상일 것 |
| 제외 조건 1 | 일정 시작 시각이 이미 지난 경우 |
| 제외 조건 2 | 사용자가 일정 브리핑 알림을 OFF로 설정한 경우 |
| 제외 조건 3 | 동일 일정에 대해 브리핑이 이미 발송된 경우 |
| 발송 주기 | 일정 1건당 1회 |
| 동시 일정 처리 | 동일 시간대 일정이 2건 이상일 경우, 각 일정에 대해 브리핑을 **모두 발송** (순차 발송, 5분 간격) |

**동시 일정 발송 순서 (동일 시간대)**

1. 외부 참석자가 있는 일정
2. 주최자가 본인인 일정
3. 참석자 수가 많은 일정
4. 캘린더 등록 순서 (빠른 것 우선)

**동시 일정 충돌 안내**

- 동일 시간대에 2건 이상 일정이 겹칠 경우, AI가 자연어로 충돌 사실을 안내한다.
- 별도 섹션이나 경고 UI가 아닌, AI 응답 본문 안에서 자연스럽게 언급한다.
- 안내 예시: "참고로, 같은 시간에 [{겹치는 일정명}] 일정도 잡혀 있어서 시간이 겹쳐요. 어느 쪽에 참석할지 미리 확인해두시면 좋겠어요."

---

#### 데이터 수집 범위 및 쿼리 조건

**[섹션 A] 일정 요약**

| 항목 | 조건 |
|------|------|
| 수집 대상 | 트리거된 캘린더 일정 단건 |
| 포함 항목 | 일정 제목, 시작/종료 시각, 장소 또는 회의 링크, 주최자, 참석자 목록, 일정 설명(메모) |
| 참석자 표시 | 최대 5명 표시, 초과 시 "외 N명" 표시 |

**[섹션 B] 관련 업무 및 자료**

| 항목 | 조건 |
|------|------|
| 수집 대상 | 일정과 연관된 플로우 업무, 게시글, 일정, 위키 등 |
| 연관 판단 기준 | 1) 일정 제목·본문 키워드 매칭, 2) 참석자 데이터 기반 (참석자가 담당자/작성자인 항목), 3) 일정 날짜와 마감일이 같은 업무 |
| 검색 범위 | 플로우 업무, 게시글, 댓글, 채팅, 위키, 캘린더 일정 |
| 상태 조건 | 모든 상태 포함 (요청, 진행중, **완료 포함** — 완료된 업무도 참고 자료가 될 수 있음) |
| 웹 검색 | 필요 시 웹 검색 병행 (외부 고객사 미팅, 기술 주제 등) |
| 정렬 기준 | 관련도순 (키워드 매칭 점수 + 참석자 연관도) |
| 포함 항목 | 항목 제목, 유형(업무/게시글/위키 등), 상태, 마감일(업무인 경우), 담당자/작성자, 프로젝트명 |
| 최대 노출 수 | 최대 5건 |
| 데이터 없을 시 | 섹션 비표시 |

**[섹션 C] 준비 필요 사항**

| 항목 | 조건 |
|------|------|
| 생성 방식 | AI가 일정 설명 + 관련 업무 기반으로 자동 생성 |
| 포함 항목 | 확인해야 할 문서/업무, 논의 예정 안건(일정 설명 기반), 행동 필요 사항 |
| 최대 노출 수 | 최대 3개 항목 |
| 데이터 없을 시 | "준비 사항이 확인되지 않아요. 일정 설명을 추가하면 더 도움이 돼요." 표시 |

---

#### 브리핑 구성 섹션

```
[일정 브리핑 구조]

1. 알림 문구
   예: "10분 후 [{일정명}] 일정이 시작돼요."

2. AI 자연어 응답 (일정 요약 + 관련 자료 + 준비 사항 통합)
   - 일정 시간, 참석자, 장소 안내
   - 동시 일정 충돌 시 자연어로 안내 (별도 UI 없음)
   - 관련 업무·게시글·위키 등을 자연어로 요약 (카드 나열이 아닌 대화체)
   - 준비 사항도 자연어로 이어서 안내
   - 관련 자료가 없으면 해당 내용 생략

3. 일정 상세 카드 (섹션 A)
   - 시간, 장소/링크, 참석자 정보를 카드 형태로 표시

4. 추천 행동 (CTA)
   - "지금 회의 링크 열기" / "관련 업무 확인하기" 등
```

**AI 자연어 응답 작성 원칙**

- 관련 자료(섹션 B)와 준비 사항(섹션 C)은 별도 제목·카드로 분리하지 않고, AI가 대화하듯 자연스럽게 설명한다.
- 누가 작성한 어떤 자료인지, 왜 참고하면 좋은지를 맥락과 함께 전달한다.
- 예시: "플로우에서 관련 자료를 찾아봤는데, 이대리님이 정리한 **주간 업무 현황**이 있고, 박개발님이 올린 **회고 요약** 게시글도 있어요. 지난주 액션 아이템이 포함되어 있으니 스탠드업 전에 한번 훑어보면 좋겠어요."

---

#### 빈 데이터 처리

| 구분 | 데이터 없음 처리 방식 |
|------|----------------------|
| 일정 정보 | 필수 — 수집 실패 시 브리핑 생성 불가 |
| 관련 자료 | AI 응답에서 관련 자료 언급 생략 (별도 안내 없음) |
| 준비 사항 | AI 응답에서 준비 사항 언급 생략, 일정 설명 추가를 권유할 수 있음 |

---

#### 예외 케이스

| 케이스 | 처리 방법 |
|--------|-----------|
| 일정 정보 수집 실패 | 브리핑 생성 취소, 에러 처리 정책 적용 |
| 일정이 생성/수집 사이에 취소됨 | 발송 취소, 사용자에게 "일정이 취소되었어요." 알림 |
| 일정 시간 변경으로 트리거 시점 충돌 | 변경된 시간 기준으로 재스케줄링 |
| 연속된 일정으로 발송 시점 겹침 | 이전 일정 브리핑 발송 후 5분 간격으로 다음 발송 |
| 동일 시간대 일정 2건 이상 | 모두 발송, 각 브리핑 본문에 "~일정과 겹칩니다" 충돌 안내 포함 |
| 일정 참석 거부 상태 | 브리핑 발송 제외 |
| 회의 링크 없음 | 장소/링크 항목에 "링크 정보 없음" 표시 |

### 2-3. 마무리 브리핑

#### 개요

하루 업무 종료 시점에 완료한 업무와 미처리 업무를 정리하고, 다음날 일정과 함께 다음 행동을 제안하는 브리핑.

---

#### 트리거 조건

| 구분 | 내용 |
|------|------|
| 트리거 유형 | 상태 판단형 (State-based) |
| 판단 조건 1 | 사용자가 설정한 퇴근 시간 이후 (기본값: 18:00) |
| 판단 조건 2 | 마지막 앱 접속 후 1시간 경과 |
| 발동 조건 | 조건 1 또는 조건 2 중 먼저 충족되는 시점 |
| 최소 발동 시각 | 12:00 이후 (오전 중 발동 방지) |
| 최대 발동 시각 | 23:00 (이후에는 발송 중단, 익일 오전 브리핑으로 대체) |
| 제외 조건 | 당일 마무리 브리핑이 이미 발송된 경우 |
| 발송 주기 | 하루 1회 |

**생성 시점:** 조건 충족 시점에 즉시 생성 (사전 준비형 아님)

---

#### 데이터 수집 범위 및 쿼리 조건

**[섹션 A] 완료 업무 요약**

| 항목 | 조건 |
|------|------|
| 수집 대상 | 사용자가 담당자인 업무 중 당일 '완료' 처리된 업무 |
| 날짜 조건 | 당일 00:00 ~ 브리핑 생성 시각 |
| 정렬 기준 | 완료 처리 시각 내림차순 |
| 포함 항목 | 업무 제목, 완료 처리 시각, 프로젝트명 |
| 최대 노출 수 | 최대 5건, 초과 시 "외 N건 완료" 표시 |
| 데이터 없을 시 | "오늘 완료한 업무가 없어요." 표시 후 섹션 유지 |

**[섹션 B] 미처리 업무 (당일 마감)**

| 항목 | 조건 |
|------|------|
| 수집 대상 | 사용자가 담당자인 업무 중 미완료 상태인 업무 |
| 마감일 조건 | 당일 마감 업무만 (지연 업무는 섹션 B-2에서 처리) |
| 상태 조건 | '완료' 제외 ('요청', '진행중' 포함) |
| 정렬 기준 | 1순위: 상태 '요청' 우선 (미착수), 2순위: 우선순위 높은 순 |
| 포함 항목 | 업무 제목, 마감일, 상태, 프로젝트명 |
| 최대 노출 수 | 최대 7건, 초과 시 "외 N건" 표시 |
| 데이터 없을 시 | 섹션 비표시 |

**[섹션 B-2] 지연 업무 요약**

| 항목 | 조건 |
|------|------|
| 수집 대상 | 사용자가 담당자인 업무 중 마감일이 지난 미완료 업무 |
| 상태 조건 | '완료' 제외 ('요청', '진행중' 포함) |
| 노출 방식 | 개별 목록 없이 카운트 요약만 표시 |
| 표시 문구 | "외 N건의 지연 업무가 있어요. 확인하기" → 플로우 업무 목록 연결 |
| CTA | "이미 완료했다면 상태 업데이트하기" |
| 데이터 없을 시 | 섹션 비표시 |

**[섹션 C] 내일 일정**

| 항목 | 조건 |
|------|------|
| 수집 대상 | 사용자가 참여자인 익일 캘린더 일정 |
| 날짜 범위 | 익일 00:00 ~ 23:59 |
| 정렬 기준 | 시작 시각 오름차순 |
| 포함 항목 | 일정 제목, 시작/종료 시각, 참석자 수, 장소/링크 |
| 최대 노출 수 | 최대 5건 |
| 데이터 없을 시 | "내일 등록된 일정이 없어요." 표시 후 섹션 유지 |

---

#### 브리핑 구성 섹션

```
[마무리 브리핑 구조]

1. 마무리 문구
   예: "오늘 하루도 수고하셨어요, {이름}님. 오늘의 업무를 정리했어요."

2. 완료 업무 요약 (섹션 A)
   - 완료 건수 강조 표시 ("오늘 N건의 업무를 완료했어요.")
   - 업무 목록 표시

3. 미처리 업무 (섹션 B)
   - 당일 마감 미처리 업무만 표시
   - 없을 경우 섹션 숨김

4. 지연 업무 요약 (섹션 B-2)
   - 지연 업무가 있을 경우 카운트만 표시
   - "외 N건의 지연 업무가 있어요. 확인하기"
   - 없을 경우 섹션 숨김

5. 내일 일정 (섹션 C)
   - 시간순 일정 목록 표시

6. 추천 행동 (CTA)
   - 미처리 업무가 있으면 "지금 바로 [업무명] 처리하기" 제안
   - 없으면 "내일 [일정명] 준비해두기" 또는 "오늘도 수고하셨어요." 표시
```

---

#### 빈 데이터 처리

| 섹션 | 데이터 없음 처리 방식 |
|------|----------------------|
| 완료 업무 요약 | "오늘 완료한 업무가 없어요." 표시, 섹션 유지 |
| 미처리 업무 | 섹션 전체 숨김 |
| 지연 업무 요약 | 지연 업무 없으면 섹션 전체 숨김 |
| 내일 일정 | "내일 등록된 일정이 없어요." 표시, 섹션 유지 |
| 전체 섹션 비어있음 | "오늘은 정리할 내용이 없어요. 내일도 화이팅!" 표시 |

---

#### 예외 케이스

| 케이스 | 처리 방법 |
|--------|-----------|
| 23:00 이전에 조건 미충족 | 발송 없이 종료, 익일 오전 브리핑에서 전날 미처리 업무 포함 |
| 사용자가 자정 전까지 앱을 계속 사용 중 | 마지막 접속 1시간 경과 조건 우선 적용 |
| 업무 데이터 수집 실패 | 수집 가능한 데이터로 부분 생성, 실패 섹션에 안내 문구 표시 |
| 퇴근 시간 미설정 | 기본값 18:00 적용 |
| 당일 오전 브리핑 미수신 상태 | 마무리 브리핑 정상 발송 (독립적으로 동작) |

## 3. 브리핑 공통 스펙

---

### 3-1. 중복 제어 로직

#### 처리 흐름

```
[브리핑 생성 시 중복 제어 처리 순서]

1. 브리핑 생성 요청 수신
2. 당일 이미 발송된 브리핑 목록 조회
3. 각 항목에 대해 중복 여부 판단
   ┌─ 동일 항목 ID (업무 ID / 일정 ID / 메시지 ID) 포함 여부 확인
   ├─ 동일 사용자 행동 요구 여부 확인
   └─ 동일 의미 정보 여부 확인 (텍스트 유사도 기반)
4. 중복 판단 결과에 따른 처리
   ┌─ 요약형 브리핑 (오전/마무리): 재정리 형태로 포함 허용
   └─ 이벤트형 브리핑 (일정): 동일 항목 제외
5. 상태 변화 예외 처리
   ┌─ 업무 상태 변경 (예: 요청→진행중, 진행중→완료) → "업데이트" 레이블 부착 후 포함
   ├─ 마감 임박 전환 → 중요도 상승 레이블 부착 후 포함
   └─ 일정 내용 변경 → 변경 내역 중심으로 재구성 후 포함
6. 중복 제어 완료 항목 기반으로 최종 브리핑 생성
```

#### 중복 판단 기준 상세

| 판단 기준 | 설명 | 예시 |
|-----------|------|------|
| 동일 ID | 동일한 업무/일정/메시지 ID를 참조 | 업무 #1234가 오전 브리핑에 이미 포함된 경우 |
| 동일 행동 요구 | 사용자에게 동일한 행동을 반복 요청 | "업무 #1234 확인하기"를 동일 시간대에 두 번 요청 |
| 동일 의미 | 문구는 다르지만 내용이 동일 | "마감 임박" vs "오늘 마감" — 동일 업무 지칭 시 중복 처리 |

#### 중복 제어 시간 범위

- 요약형 브리핑 간: 당일 내 발송된 전체 브리핑 기준
- 이벤트형 브리핑 간: 동일 이벤트(일정 ID 기준) 당 1회

---

### 3-2. 브리핑 스타일 적용

#### 스타일 정의

| 스타일 | 설명 | 어조 특징 | 문장 길이 |
|--------|------|-----------|-----------|
| 핵심형 | 핵심 정보만 간결하게 요약 | 명사형 종결, 리스트 중심 | 짧음 (1~2줄) |
| 근거형 | 정보에 이유와 분석을 함께 제공 | 설명형, 이유 포함 | 중간 (2~4줄) |
| 대화형 | 친근하고 자연스러운 대화 방식 | 구어체, 이모지 포함 가능 | 중간 (2~3줄) |
| 액션형 | 즉각 실행을 유도하는 지시형 | 동사 시작, 명령형 | 짧음 (1~2줄) |

#### 스타일별 동일 데이터 출력 예시

> 데이터: 업무 "Q2 보고서 작성", 마감: 오늘 18:00, 상태: 진행중

| 스타일 | 출력 예시 |
|--------|-----------|
| 핵심형 | Q2 보고서 작성 — 오늘 18:00 마감 |
| 근거형 | 'Q2 보고서 작성'이 오늘 오후 6시에 마감돼요. 현재 진행 중 상태로, 제출 전 최종 검토가 필요해요. |
| 대화형 | 오늘 마감인 'Q2 보고서 작성' 잊지 않으셨죠? 아직 진행 중이니 서두르는 게 좋을 것 같아요! |
| 액션형 | 지금 바로 Q2 보고서 작성 완료하세요. 마감까지 남은 시간을 확인하세요. |

#### 스타일 적용 범위

- 모든 브리핑 유형(오전/일정/마무리)에 동일하게 적용
- 사용자가 스타일 미설정 시 기본값: **핵심형**
- 스타일 변경 즉시 다음 브리핑부터 적용 (당일 이미 발송된 브리핑은 변경 불가)

### 3-3. 전달 방식

#### 전달 채널

| 채널 | 설명 | 적용 브리핑 |
|------|------|-------------|
| 푸시 알림 | 디바이스 푸시 알림으로 브리핑 도착 알림 | 오전 브리핑, 일정 브리핑 |
| AI 비서 홈 화면 | AI 비서 탭 진입 시 최신 브리핑 표시 | 전체 브리핑 |
| 인앱 배너 | 앱 사용 중 상단 배너로 알림 표시 | 마무리 브리핑 (사용 중인 경우) |

#### 전달 동작 상세

**푸시 알림**

- 푸시 알림 제목: 브리핑 유형명 (예: "오전 브리핑이 도착했어요")
- 푸시 알림 본문: 브리핑 핵심 요약 1줄 (최대 100자)
- 클릭 시 동작: AI 비서 홈 화면으로 이동 후 해당 브리핑 펼침
- 알림 권한 없을 시: AI 비서 홈 화면 뱃지만 표시

**AI 비서 홈 화면**

- 최신 브리핑을 홈 화면 최상단에 카드 형태로 표시
- 읽지 않은 브리핑: 강조 표시 (뱃지 또는 컬러 처리)
- 읽은 브리핑: 히스토리 영역으로 이동


#### 발송 우선순위 (동시 발생 시)

우선순위 높은 순으로 푸시 알림 순차 발송. 동시 발생 시 최대 2건까지 합산 알림 가능.

1. 일정 브리핑
2. 오전 브리핑 / 마무리 브리핑
3. (Phase 2 이후) 변화 감지 브리핑

## 6. 상태 관리

---

### 6-1. 브리핑 읽음/안읽음 상태

#### 상태 정의

| 상태 | 설명 | 전환 조건 |
|------|------|-----------|
| 안읽음 (UNREAD) | 사용자가 아직 열람하지 않은 브리핑 | 브리핑 생성 직후 기본 상태 |
| 읽음 (READ) | 사용자가 열람한 브리핑 | 브리핑 카드 상세 화면 진입 시 자동 전환 |

#### 상태 표시 방식

- AI 비서 홈 화면 뱃지: 안읽음 브리핑 수 표시
- 브리핑 카드: 안읽음 상태일 경우 강조 처리 (색상 또는 점 인디케이터)
- 앱 아이콘 뱃지: 안읽음 브리핑 수 표시 (OS 설정 허용 시)

#### 읽음 처리 조건

- 브리핑 상세 화면에 진입하면 즉시 읽음 처리
- 푸시 알림 클릭 → 앱 진입 시 해당 브리핑 자동 열람 → 읽음 처리
- 읽음 상태는 디바이스 간 동기화 (서버 기준)

---

### 6-2. 브리핑 히스토리 보관

| 항목 | 정책 |
|------|------|
| 보관 기간 | 최근 30일 |
| 보관 단위 | 브리핑 1건 = 1개 레코드 (생성 시각, 유형, 내용, 읽음 여부 포함) |
| 만료 처리 | 30일 경과 시 자동 삭제 |
| 삭제 알림 | 없음 (별도 안내 불필요) |
| 히스토리 조회 | AI 비서 홈 화면 내 "지난 브리핑" 목록에서 확인 가능 |
| 정렬 | 생성 시각 내림차순 (최신순) |
| 표시 제한 | 최대 100건 표시 (30일 내 100건 초과 시 최신 100건만 표시) |

## 7. 에러 처리 정책

---

### 7-1. 데이터 수집 실패

| 실패 유형 | 처리 방식 | 사용자 노출 |
|-----------|-----------|-------------|
| 단일 데이터 소스 실패 (캘린더 OR 업무 OR 메시지 중 1개) | 해당 섹션 제외하고 나머지 데이터로 브리핑 부분 생성 | 해당 섹션 위치에 "정보를 불러오지 못했어요. [섹션명] 정보가 포함되지 않았어요." 인라인 표시 |
| 2개 이상 데이터 소스 실패 | 수집 가능한 데이터로만 브리핑 생성 시도 | 브리핑 상단에 "일부 정보를 불러오지 못해 내용이 불완전할 수 있어요." 배너 표시 |
| 전체 데이터 소스 실패 | 브리핑 생성 불가, 생성 실패로 전환 | 생성 실패 처리 (7-2 참고) |
| 데이터 수집 타임아웃 | 3초 이내 응답 없을 경우 해당 소스 실패로 처리 | 위 단일 실패 처리와 동일 |

**재시도 정책 (데이터 수집 단계)**

- 자동 재시도: 1회 (30초 후)
- 재시도 후에도 실패 시 해당 소스 실패로 최종 처리

---

### 7-2. 브리핑 생성 실패

| 실패 유형 | 처리 방식 | 사용자 노출 |
|-----------|-----------|-------------|
| AI 생성 오류 | 재시도 1회 실행 (즉시) | 재시도 중 로딩 표시 |
| 재시도 후 생성 실패 | 브리핑 생성 실패 확정 | "브리핑을 만드는 데 문제가 생겼어요. 다시 시도해주세요." + [다시 시도] 버튼 표시 |
| 수동 재시도 | 사용자가 [다시 시도] 버튼 클릭 시 생성 재요청 | 로딩 표시 후 성공/실패 결과 표시 |
| 수동 재시도 실패 | 2회 수동 재시도 후에도 실패 시 | "일시적인 문제가 발생했어요. 잠시 후 다시 확인해주세요." 안내 후 버튼 제거 |

**생성 실패 시 브리핑 히스토리 기록**

- 생성 실패도 히스토리에 기록 (상태: 생성 실패, 내용 없음)
- 사용자가 히스토리에서 해당 시점 브리핑 확인 시 실패 상태 표시

---

### 7-3. 브리핑 전달 실패

| 실패 유형 | 처리 방식 | 사용자 노출 |
|-----------|-----------|-------------|
| 푸시 알림 발송 실패 | 푸시 발송 실패 시 AI 비서 홈 뱃지만 표시 | 별도 안내 없음 (홈 진입 시 브리핑 확인 가능) |
| 인앱 배너 표시 실패 | 배너 표시 생략, 홈 화면에서 확인 가능 | 별도 안내 없음 |
| 브리핑 로딩 실패 (홈 화면 진입 시) | 3초 이내 응답 없을 경우 재시도 1회 | 로딩 인디케이터 표시 후 실패 시 "브리핑을 불러오지 못했어요. [다시 시도]" 표시 |

---

### 7-4. 에러 메시지 가이드라인

- 에러 메시지는 기술적 표현을 사용하지 않음
- 사용자가 취할 수 있는 행동을 항상 포함
- 예시:
  - X "HTTP 500 오류가 발생했습니다."
  - O "브리핑을 만드는 데 문제가 생겼어요. 다시 시도해주세요."

   ├─ 중복 판단 (ID / 행동 / 의미 기준)
   ├─ 요약형: 재정리 허용, 이벤트형: 중복 제외
   └─ 상태 변화 항목: 업데이트 레이블 부착

5. 브리핑 생성
   ├─ 필터링된 데이터를 브리핑 구조에 매핑
   ├─ 사용자 스타일 설정 적용
   └─ AI 생성 (준비 사항 등 자동 생성 섹션 포함)

6. 전달
   ├─ 푸시 알림 발송 (설정 ON인 경우)
   ├─ AI 비서 홈 화면 업데이트
   └─ 히스토리 레코드 저장
