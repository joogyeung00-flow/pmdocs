## 2. 논의사항

### 2-1. V1 출시 전 결정 필요 (기한: ~2026-04-20)

#### [D-1] 브리핑 발송 시간 A/B 테스트 여부 (OI-002)
- **현재**: 기본값 오전 08:00, 사용자 설정 가능 (07:00~09:30)
- **쟁점**: 오전 8시 vs 9시 그룹 비교 실험을 MVP 단계에서 진행할 것인가?
- **고려사항**:
  - A/B 테스트 인프라 구축 비용 vs MVP 일정 영향
  - 사용자가 직접 시간 설정 가능하므로, 베타 기간 사용 패턴 데이터 수집으로 대체 가능한지
- **제안**: MVP에서는 기본값 08:00으로 출시, 오픈 베타 기간 중 설정 변경 패턴 분석 후 V2에서 A/B 테스트 진행

#### [D-2] 폴백 발동 시 사용자 고지 방식 (OI-007)
- **현재**: 폴백 발동율 목표 3% 미만이지만, 발동 시 사용자에게 AI 실패 사실을 명시할지 미결정
- **선택지**:
  - A: 명시적 고지 ("AI 생성에 실패했어요. 다시 시도해주세요")
  - B: 조용히 재시도 후 최종 실패 시에만 고지
  - C: 폴백 콘텐츠(간소화 버전) 제공 + "더 자세한 브리핑 보기" 재생성 버튼
- **제안**: C안 — 사용자 경험 끊김 최소화하면서도 투명성 확보

#### [D-3] PRD 내 캐릭터 관련 정보 정합성
- **현황**: PRD v2.3에서 "캐릭터 기획 전면 제거, 통일 AI 아바타 도입"이라고 변경 이력에 명시되어 있으나, 본문에 캐릭터 3종(플로키/펭귀니/거북이) 정의가 여전히 남아있음 (§5-8)
  - 온보딩은 4단계(캐릭터 선택 단계 제거됨)로 정리되었음
  - 설정 화면에는 "AI 캐릭터" 서브섹션에 3종 캐릭터 칩이 남아있음
- **쟁점**: 통일 아바타 1종으로 갈 것인지, 캐릭터 3종을 유지할 것인지 최종 확정 필요
- **영향 범위**: 온보딩 플로우, 설정 화면, 사이드바, 프로필 카드, 인사 문구 등 전반

#### [D-4] Phase 1 스펙(FRD) vs PRD 간 데이터 범위 차이
- **PRD (FLOW-001-ai-smart-secretary-prd.md)**:
  - 오전 브리핑 일정 최대 4건, 마감 업무 최대 5건
  - 마감 업무: 당일(D+0) 또는 익일(D+1)
- **FRD (FLOW-001-phase1-spec.md)**:
  - 오전 브리핑 일정 제한 없음(전체 표시), 업무 최대 10건
  - 마감 업무: 당일 + 1~2일 지연 업무 + 마감일 미설정 업무 포함
  - 지연 업무 리마인드(섹션 B-2) 별도 추가
  - 확인해야할 업무(섹션 B-1) 최대 5건 별도
- **쟁점**: 두 문서 중 어느 것을 정본(source of truth)으로 할 것인가?
- **제안**: PRD를 방향성/원칙, FRD를 구현 상세 정본으로 역할 분리하되, 핵심 수치(최대 건수 등)는 FRD 기준으로 통일

### 2-2. V1 출시 전 결정 필요 (기한: ~2026-05-01)

#### [D-5] 주말·공휴일 브리핑 정책 (OI-005)
- **현재**: 기본값 OFF (주말 미발송), V2에서 발송 허용 설정 추가 예정
- **쟁점**: 주말 OFF가 월요일 출근 시 정보 과부하를 만들 수 있음
  - 월요일 브리핑에 주말 동안의 변화를 모두 포함하면 브리핑 길이 과다
- **제안**: 월요일 브리핑은 "주말 요약" 모드로 확장 — 금요일 퇴근 이후 ~ 월요일 출근까지의 변화를 별도 섹션으로 제공

#### [D-6] 퇴근 리포트 트리거 시각 설정 가능 여부 (OI-009)
- **현재**: 18:00 고정 + 마지막 접속 1시간 뒤
- **쟁점**:
  - 야근이 일상인 팀: 18:00 기준이 너무 이름 → 불필요한 알림
  - 유연근무제 조직: 출퇴근 시간이 개인마다 다름
- **선택지**:
  - A: V1에서 18:00 고정 유지 (단순화)
  - B: V1에서 설정 가능하게 (18:00 / 19:00 / 20:00 / 21:00)
- **제안**: V1에서는 A안(18:00 고정)으로 출시, 베타 피드백 수집 후 V2에서 설정 추가

### 2-4. 아키텍처·기술 논의 사항

#### [T-1] LLM 비용 최적화 전략
- **KPI 목표**: 실제 토큰 비용/MAU $1 미만
- **논의 필요**:
  - 브리핑 3개 섹션을 하나의 LLM 호출로 생성할지, 섹션별 분리 호출할지
  - 캐시 정책: 동일 사용자 동일 날짜 브리핑 캐시 → 데이터 변경 시 재생성 기준
  - 브리핑 스타일 4종에 따른 프롬프트 차이가 토큰 비용에 미치는 영향

#### [T-2] 데이터 수집 파이프라인 설계
- 오전 브리핑: 발송 30분 전 수집 시작 → 대규모 사용자 동시 수집 시 부하
- 퇴근 리포트: 행동 기반 트리거 → 서버 측 접속 로그 폴링 주기 결정 필요
- 일정 리마인더: 일정별 개별 트리거 → 스케줄링 시스템 설계

#### [T-3] Prompt Guard 정책 구체화
- 자동 마스킹 대상은 정의되어 있으나, 마스킹 후 LLM 요약 품질 검증 필요
- 댓글 원문 비노출 정책 → LLM에 원문 전달 후 요약만 출력하는 방식의 안전성 검증

### 2-5. 디자인 핸드오프 확인사항

#### [UX-1] 온보딩 4단계 최종 확정
- 변경 이력: 2단계 → 5단계 → 4단계 (캐릭터 선택 제거)
- 현재 4단계: 브리핑 스타일 → 오전 브리핑 → 일정 준비 → 퇴근 리포트
- 확인 필요: 스텝 번호 혼선 (PRD 본문에 Step 4가 Step 2 앞에 위치하는 등 순서 불일치)

#### [UX-2] 브리핑 스타일 정의 통일
- **온보딩 (§5-1)**: 4종 — 요약형 / 분석형 / 친근형 / 지시형
- **FRD**: 4종 — 핵심형 / 근거형 / 대화형 / 액션형
- **대화 스타일 (§5-8)**: 7종 — 기본적인 / 진중한 / 친근한 / 솔직한 / 독특한 / 간결한 / 까칠한
- **쟁점**: "브리핑 스타일"과 "대화 스타일"이 별개 개념인지, 통합할 것인지
- **제안**: 브리핑 스타일 4종(콘텐츠 구조)과 대화 스타일 7종(말투 톤)을 분리 유지하되, 명칭을 PRD 기준으로 통일
